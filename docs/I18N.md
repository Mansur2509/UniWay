# Internationalization

## Supported locales

EduVerse currently supports:

- `en` — English
- `ru` — Русский
- `uz-Latn` — O‘zbekcha
- `uz-Cyrl` — Ўзбекча

English is the default locale. It provides deterministic initial rendering and acts as the source dictionary for compile-time key coverage.

## Structure

```text
frontend/src/shared/i18n/
  config.ts                 locale codes and metadata
  types.ts                  translation key and dictionary types
  provider.tsx              state, persistence, interpolation, useI18n()
  dictionaries/
    en.ts
    ru.ts
    uz-latn.ts
    uz-cyrl.ts
    beta-preview.en.ts
    beta-preview.ru.ts
    beta-preview.uz-latn.ts
    beta-preview.uz-cyrl.ts
    onboarding.en.ts
    onboarding.ru.ts
    onboarding.uz-latn.ts
    onboarding.uz-cyrl.ts
```

The language switcher persists the selected locale in localStorage under `eduverse.locale`. The provider updates the root document `lang` and `dir` attributes.

Large coherent UI domains may use locale-specific dictionary fragments and spread them into the main locale dictionary. Fragment filenames must preserve the domain and locale pattern, and their keys remain part of the same global typed namespace. The parity script reads both the base and registered beta-preview fragments.

## Using translations

Client components call:

```tsx
const { t } = useI18n();

return <Button>{t("common.actions.continue")}</Button>;
```

Simple interpolation uses named placeholders:

```tsx
t("shell.signedInAs", { role: t("roles.student") });
```

## Key naming

Keys use dot-separated domain namespaces:

```text
navigation.dashboard
auth.login.title
profile.intendedMajor
dashboard.highlight1.detail
common.error.generic
a11y.logout
```

Rules:

1. Start with the owning UI or domain namespace.
2. Name the meaning, not the English wording.
3. Use camelCase inside a segment.
4. Keep reusable actions, errors, empty states, and accessibility text under `common.*` or `a11y.*`.
5. Do not build sentences by concatenating translated fragments.
6. Use named interpolation for dynamic values.

## Adding a language

1. Add the locale code to `supportedLocales` in `config.ts`.
2. Add metadata with label, native label, and direction.
3. Create a dictionary implementing `TranslationDictionary`.
4. Register it in `dictionaries/index.ts`.
5. Run the local dictionary-key parity check and available TypeScript checks.
6. Review the language in context with a fluent speaker before release.

Run the dependency-free parity check with:

```powershell
cd frontend
node scripts/check-i18n.mjs
```

## No hardcoded strings

New or modified user-facing text must use a translation key. This includes visible labels, buttons, errors, empty states, loading messages, tooltips, `aria-label` values, and icon-only button names.

Exceptions are product names, user-generated values, backend data, and temporary developer-only diagnostics. Server metadata remains English until locale-aware routing is introduced.
