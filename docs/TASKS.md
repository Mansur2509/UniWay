- [x] **DATA-LOADING-RESILIENCE-001** Fix the reported "broken" data loading on Universities, Event Map, Essays, and Applications. Root-caused empirically (not assumed): the endpoints are healthy — all four return `200` with valid JSON when the backend is warm, CORS is correctly configured (Vercel origin whitelisted, verified via `OPTIONS` preflight), the Events query already has `select_related` + a count annotation (no N+1), and Supabase RLS does not block Django (server-side connection returns full data). The real causes were (1) **Render free-tier cold start** — measured live twice at 64s and >90s for the first request after idle, vs ~0.8s warm — combined with (2) **a frontend fetch layer that had no timeout** (so a cold start hung as an infinite spinner), and (3) **no React error boundary plus a `t()` helper that threw on a missing/dynamic key**, which unmounted the whole route into a blank page (the Essays "blank, no feedback" symptom). Fixes: added an `AbortController` timeout (`REQUEST_TIMEOUT_MS=90s`) to the shared `apiRequest` that converts timeouts/network failures into a typed, retryable `ApiError` instead of hanging; hardened i18n `t()` to fall back to the key instead of throwing; added `app/error.tsx` + `app/global-error.tsx` so no render error can ever produce a blank page; added a `useSlowLoad` hook + `LoadingNotice` "the server may be waking up…" message (after 5s) on the auth/session gate and all four data screens; added a free `.github/workflows/keepalive.yml` that pings `/api/v1/health/` every 10 min to keep the service warm, and documented the cold-start analysis + the Render Starter (~$7/mo, eliminates spin-down) recommendation in `docs/OPERATIONS.md`. Verified: typecheck clean, ESLint clean, i18n parity (1421 keys x 4 locales), Next.js build clean (25 routes), 156/156 backend tests still pass (no backend code changed). Live browser QA (headless preview against the local backend, fixed build): Dashboard + Universities + Essays + Events + Applications all render with real data, **zero console errors or warnings** across the whole session; killing the backend and re-fetching Universities showed the clean "We could not load…" error + working "Try again" (no blank, no infinite spinner) and recovered on retry; the "Include demo data" toggle loads in both states. Production repeated-load timing (warm, 10 runs each, 0 failures): Universities avg 4.1s, Events 3.6s, Essays 3.3s, Applications 3.3s (absolute numbers inflated by the sandbox's network path + a just-woken service; direct single hits and the local browser were ~1-2s/sub-second). Cold-start cannot be eliminated without deploying the fix + keep-alive or upgrading Render; the deployed-cold-start browser test is documented for the user to run post-deploy.

- [x] **UNIVERSITIES-API-PATH-HOTFIX-001** Fix the production `/universities` page showing "We could not load the university catalog." Root-caused empirically by inspecting the deployed Vercel bundle: `NEXT_PUBLIC_API_BASE_URL` was set to the bare backend origin `https://eduverse-vvw2.onrender.com` with no `/api/v1` suffix (the bundle literally contained `apiBaseUrl:"https://eduverse-vvw2.onrender.com"` and no `/api/v1` anywhere). Because the DRF router (universities, exams, profiles, subscriptions) is mounted under `/api/v1/`, every `base: "api"` call hit `<origin>/universities/` and 404'd, while auth/events/roadmap/essays/applications kept working because `env.ts` rebuilds those from `new URL(apiBaseUrl).origin` (which discards the path). This is the same `GET /universities/` 404 seen earlier in the Render logs — it was a real frontend misconfiguration, not bot noise. Fix (code-side, so it self-heals regardless of the env value and is defensive against recurrence): `env.ts` now normalizes `apiBaseUrl` — a value already ending in `/api/v1` is kept as-is, anything else is rewritten to `new URL(value).origin + "/api/v1"`. Added dev-only diagnostics to `getUniversitiesRequest` (final request URL, status, message, response body preview; never logs tokens). Response normalization (array vs DRF `{results}`) and the empty-vs-error states were already handled by `normalizePaginatedResponse` + the screen. Verified locally with `.env.local` deliberately set to the broken bare origin: DevTools/preview Network showed the catalog request resolve to `https://eduverse-vvw2.onrender.com/api/v1/universities/` → 200 (not the bare `/universities/`), catalog loaded real data (Stanford, Cambridge, ...), no error state, zero console errors. typecheck/lint/build clean.
